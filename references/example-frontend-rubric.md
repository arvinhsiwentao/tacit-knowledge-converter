# 參考範例：Anthropic Frontend Development Rubric

這份 Rubric 來自 Anthropic 對前端產出品質的評估標準。
它是一個經過實戰驗證的範例，展示了一份好的 Rubric 應該長什麼樣。

讀這份範例時，注意以下幾件事：
- 每個維度都有明確的定義，不是模糊的形容詞
- 有「什麼叫好」也有「什麼叫不好」的具體描述
- 維度之間有刻意的權重分配（Design Quality 和 Originality 的權重高於 Craft 和 Functionality）
- 權重分配背後有明確的理由（模型本身在 Craft/Functionality 上已經夠好，瓶頸在創意和美感）

---

## 維度定義

### 1. Typography（字體）
- 選用美觀、獨特、有個性的字體
- 避免 Arial、Inter 等通用字體
- Display font 搭配 Body font，形成層次感
- **重點：** 字體選擇本身就是設計決策，不是預設值

### 2. Color & Theme（色彩與主題）
- 承諾一個連貫的美學方向
- 使用 CSS variables 確保一致性
- 主色搭配銳利的強調色，優於平均分配的膽怯配色
- **重點：** 配色要有態度，不是安全牌

### 3. Motion（動效）
- 使用動畫做效果與微互動
- HTML 優先用 CSS-only；React 可用 Motion library
- 聚焦高影響力時刻：一個精心編排的頁面載入（staggered reveals）> 零散的微互動
- Scroll-triggering 和 hover states 要有驚喜感
- **重點：** 少而精，不是到處撒動畫

### 4. Spatial Composition（空間構成）
- 非預期的版面配置
- 不對稱、重疊、對角線流動、打破格線的元素
- 大量留白 或 受控的密度
- **重點：** 構圖本身是表達，不只是排列

### 5. Backgrounds & Visual Details（背景與視覺細節）
- 營造氛圍與深度，不是預設純色背景
- 加入與整體美學匹配的語境效果與紋理
- 漸層網格、噪點紋理、幾何圖案、多層透明度、戲劇性陰影、裝飾性邊框、自定義游標、顆粒感覆蓋
- **重點：** 背景不是空白，是設計的一部分

### 6. Design Quality（設計品質）｜高權重
- 設計是否感覺像一個連貫的整體，而非零件的拼湊？
- 色彩、字體、版面、圖像和細節是否共同創造出獨特的氛圍與識別度？
- **重點：** 整體感。各部分是否服務於同一個設計意圖？

### 7. Originality（原創性）｜高權重
- 是否有自定義的設計決策，還是模板版面、函式庫預設、AI 生成模式？
- 人類設計師應該能辨識出刻意的創意選擇
- 未修改的素材元件、或 AI 生成的典型模式（例如紫色漸層疊白色卡片）在這個維度直接不及格
- **重點：** 有沒有「人做的決定」的痕跡

### 8. Craft（工藝）｜標準權重
- 技術執行：字體層次、間距一致性、色彩和諧、對比度
- 這是能力檢查，不是創意檢查
- 大多數合理的實作預設就能過關；不及格代表基本功壞了
- **重點：** 底線檢查，不是差異化因素

### 9. Functionality（功能性）｜標準權重
- 與美學無關的可用性
- 使用者能否理解介面做什麼、找到主要操作、完成任務而不需要猜？
- **重點：** 能不能用，跟好不好看是兩件事

---

## 權重設計的邏輯

Anthropic 刻意提高 Design Quality 和 Originality 的權重，降低 Craft 和 Functionality 的權重。

原因：
- 模型在 Craft 和 Functionality 上本來就表現不錯（技術能力自然就有）
- 但在 Design Quality 和 Originality 上容易產出平庸的「AI slop」
- 透過加重這兩個維度，推動模型承擔更多美學風險

這個設計決策本身就是一個好的 Rubric 最重要的特徵：
**權重反映的是「瓶頸在哪裡」，不是「什麼都一樣重要」。**

---

## 校準方法

- 使用 few-shot examples 搭配詳細的評分分解來校準 Evaluator
- 確保 Evaluator 的判斷與個人偏好對齊
- 減少跨迭代的分數漂移

---

## 作為 Rubric 範本的啟示

當你為自己的領域建立 Rubric 時，參考這份範例的幾個特徵：

1. **維度要具體到可判斷**——不是「設計好不好」，而是拆成 Typography、Color、Motion 等可獨立評估的面向
2. **每個維度要有「什麼叫不好」的反面描述**——讓 Evaluator 知道紅線在哪
3. **權重要反映瓶頸**——不是所有維度等權重，而是根據「哪裡最容易出問題」來分配
4. **要有校準機制**——few-shot examples + 評分分解，讓 Evaluator 的判斷穩定
