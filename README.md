# Tacit Knowledge Converter

把腦中模糊的「我覺得這個好」變成 AI 可以執行的評分標準。

---

## 它在解決什麼問題

當你叫 Agent 跑長任務時，它需要一個東西判斷「做完了沒」。

寫程式的人很爽——`unit test pass`、`lint clean`、`response < 0.12s`，這些都是 AI 自己可以驗證的標準。所以 Claude Code、Codex、Hermes 上面 `/goal` 跑得起來。

但大部分的白領工作不是這樣：

- 你的文案寫得好不好？
- UI 設計有沒有品味？
- 影片有沒有節奏感？
- 簡報有沒有說服力？

這些沒有 pass / fail。沒有一個指令可以告訴你「對，這個有 90 分」。

於是 AI 跑你的工作時會出現兩種失敗：

1. **草草收工**——AI 不知道什麼叫「好」，所以做兩三輪就說「我想我已經完成了」
2. **平庸收斂**——AI 跑出來的東西「不算錯但毫無記憶點」，因為沒人告訴它你要的標準是什麼

這個 skill 透過蘇格拉底式追問，把你腦中說不清楚的判準挖出來，整理成具體的 Rubric。

---

## Rubric 為什麼有用

Rubric 是「品味」跟「機器執行」之間的橋。

它讓你不需要每三分鐘跑回去檢查 AI——因為 Evaluator 拿著你的 Rubric，可以在每一輪幫你守住標準。AI 在你睡覺的時候也不會偷懶，因為它知道沒過你的標準就不能停。

寫 Rubric 真正的價值不只是給 AI 看。**寫的過程，會逼你把腦中模糊的「我覺得這個好」externalize 成文字。** 一旦變成文字，就可以被覆用、被改進、被傳承。

---

## 它輸出什麼

每次跑完會給你兩個檔案：

- **Markdown Rubric**——給人讀、討論、調整
- **JSON Rubric**——直接接到 Agent 的 Evaluator prompt

每個維度都有 **1-5 分的具體可掃描描述**（不是「品質很好」這種抽象形容詞，而是「看到 X 特徵就給 Y 分」這種錨點）。

---

## 怎麼用

```
/tacit-knowledge-converter
```

接下來 skill 會帶你跑 5 個步驟：

1. 確認領域與任務（你要評估什麼產出？）
2. 建立 Foundation（業界 + 領域專家的標準）
3. 疊入你的個人品味（蘇格拉底追問把判準挖出來）
4. 展開 1-5 分評分描述
5. 輸出 Markdown + JSON

跑完後，把 JSON 貼進你的 `/goal` prompt：

```
你要做 [任務]。

在 evaluator 沒給 ≥4 分之前不要停。
Rubric 在這裡：[paste JSON]
```

---

## 適合誰用

- **寫作者**：定義什麼叫好的文章 / 貼文 / 腳本
- **設計師**：定義什麼叫好的 UI / UX
- **PM**：定義什麼叫好的 PRD / 產品設計
- **內容創作者**：定義什麼叫好的影片 / Reels / 縮圖
- **教育者**：定義什麼叫好的 Essay / 報告
- **面試官**：定義什麼叫好的候選人表現

只要你的工作有「品味」成分但說不清楚標準是什麼——這個 skill 就是給你用的。

## Rubric 不是定稿

第一輪跑完，問自己：**Evaluator 給的分數，跟你眼睛看的一致嗎？**

不一致——不是 AI 錯，是你的 Rubric 還沒抓到你真正的 taste。回去改 Rubric，再跑一輪。

跑三、四輪，你會發現你心裡那個「我覺得什麼叫好」，慢慢地一條一條被你自己寫了出來。
